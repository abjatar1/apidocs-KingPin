KingPin backend

REST API for teams, players & matches

This service exposes endpoints to manage accounts, teams, players and bowling matches. All data is stored in MongoDB through a simple in-memory cache and persisted with db.saveAll().

Express + Node.js
Auth: password in request body
JSON over HTTP
Mongo-backed data store
Quick start
npm install
node server.js

GET / -> "Hello World!"

Schemas

Backing shapes for users, teams, players & matches

These are the informal JSON structures used internally. Fields marked with /* internal */ are not exposed on public responses.

Core objects

UserSchema

Stored as users[id] 
{
  userId: string,
  email: string,
  password: string,        // bcrypt hash
  username: string,
  displayName: string,
  teamIds: { [teamId: string]: true },
  resetToken: string | null,
  resetTokenExpiry: number | null
}

TeamSchema

Stored as teams[teamId] 
{
  teamId: string,
  ownerId: string,  // userId
  displayName: string,
  playerIds: { [playerId: string]: true },
  awardsList: string[],
  matchIds: { [matchId: string]: true }
}

PlayerSchema

Stored as players[playerId] 
{
  playerId: string,
  displayName: string,
  teamId: string,
  matchIds: { [matchId: string]: true },
  graduationYear: number | null
}

MatchSchema

Stored as matches[matchId] 
{
  matchId: string,
  teamId: string,
  date: number,                 // Unix timestamp (ms)
  opposingTeamName: string,
  comment: string,
  perPlayerData: {
    [playerId: string]: {
      games: {
        1: { Wood: number, Score: number, isVarsity: boolean },
        2: { Wood: number, Score: number, isVarsity: boolean },
        3: { Wood: number, Score: number, isVarsity: boolean }
      }
    }
  }
}

Auth & Accounts

Account lifecycle & profile management

Authentication is done via email + password login. For protected routes, the current password is sent in the JSON body (password or currentPassword) and validated with requireValidPassword.

Create account

POST
/accounts

Registers a new user and sends a welcome email. Password is stored as a bcrypt hash.

email password (≥ 8 chars) username displayName (optional) 
POST /accounts
{
  "email": "user@example.com",
  "password": "super-secure-password",
  "username": "nathan",
  "displayName": "Nathan"
}

201 Created
{
  "user": {
    "userId": "uuid",
    "email": "user@example.com",
    "username": "nathan",
    "displayName": "Nathan",
    "teamIds": {}
  }
}

Login

POST
/auth/login

Basic UI login. Returns a boolean flag and public user info; does not create a session or token.

email password 
POST /auth/login
{
  "email": "user@example.com",
  "password": "super-secure-password"
}

200 OK
{ "ok": true, "user": { ...publicUser } }

Get account (public view)

GET
/accounts/:userId

Returns the public view of a user (no password or reset token fields).

Public

Change password

POST
/accounts/:userId/password

Changes the user password after validating the current one.

requireValidPassword currentPassword newPassword (≥ 8) 
POST /accounts/:userId/password
{
  "currentPassword": "old-password",
  "newPassword": "new-super-secure"
}

Change display name

POST
/accounts/:userId/displayName

Updates the user-facing display name for this account.

requireValidPassword password displayName

Delete account (API)

DELETE
/accounts/:userId

Deletes the user and cascades deletion to all owned teams, players, and matches.

requireValidPassword password Cascade delete teams/matches

Delete account (email link)

GET
/emailDeleteAccount/:userId/:password

HTML endpoint linked from email. If the link is valid, permanently deletes the account and related data.

HTML response One-click deletion

Teams

Per-user team management

Teams are owned by a single user (ownerId). All read/write endpoints validate ownership using ensureUserAndTeam.

Create team

POST
/users/:userId/teams

Creates a new team owned by the given user and attaches it to user.teamIds.

requireValidPassword password displayName (optional)

List teams

GET
/users/:userId/teams

Returns all teams owned by this user as public team objects.

Get team

GET
/users/:userId/teams/:teamId

Fetches a single team if it belongs to the given user.

Update team

PUT
/users/:userId/teams/:teamId

Updates the team display name. Ownership is required.

requireValidPassword password displayName

Delete team

DELETE
/users/:userId/teams/:teamId

Deletes the team and cascades to all players and matches belonging to it.

requireValidPassword deleteTeamCascade()

Players

Roster management per team

Players are scoped to a single team via teamId. They also keep a matchIds map for future per-match linkage.

Create player

POST
/users/:userId/teams/:teamId/players

Adds a new player to the specified team.

requireValidPassword password displayName graduationYear (1900–2100)

List players

GET
/users/:userId/teams/:teamId/players

Returns all players in the team, as public player objects.

Get player

GET
/users/:userId/teams/:teamId/players/:playerId

Fetches a single player, ensuring they belong to the requested team.

Update player

PUT
/users/:userId/teams/:teamId/players/:playerId

Updates a player's display name and/or graduation year.

requireValidPassword displayName graduationYear

Delete player

DELETE
/users/:userId/teams/:teamId/players/:playerId

Removes the player from the team and from all matches' perPlayerData.

requireValidPassword cleanup perPlayerData

Matches

Match metadata & per-team lists

Matches are created for a team and store date, opposing team name, a free-form comment, and per-player game data.

Create match

POST
/users/:userId/teams/:teamId/matches

Creates a new match and initializes per-player game slots (1–3) with default values.

requireValidPassword date (Unix ms, optional) opposingTeamName comment

List matches

GET
/users/:userId/teams/:teamId/matches

Returns all matches for the team as stored in matches.

Get match

GET
/users/:userId/teams/:teamId/matches/:matchId

Fetch a single match, ensuring it belongs to the specified team.

Update match

PUT
/users/:userId/teams/:teamId/matches/:matchId

Updates match metadata: date, opposing team name, comment.

requireValidPassword date opposingTeamName comment

Delete match

DELETE
/users/:userId/teams/:teamId/matches/:matchId

Deletes a match and removes its id from team.matchIds.

requireValidPassword team.matchIds cleanup

Per-player match data

Game-level stats per player

Each match maintains a perPlayerData object keyed by playerId. Each player has 3 game slots, indexed 1–3.

Update a player game

PUT
/users/:userId/teams/:teamId/matches/:matchId/players/:playerId/games/:gameIndex

Sets the stats for a single game (1–3) for a given player in a given match.

requireValidPassword gameIndex ∈ {1,2,3} Wood (number) Score (number) isVarsity (boolean) 
PUT /users/:userId/teams/:teamId/matches/:matchId/players/:playerId/games/1
{
  "password": "current-password",
  "Wood": 10,
  "Score": 178,
  "isVarsity": true
}

Get per-player data

GET
/users/:userId/teams/:teamId/matches/:matchId/players/:playerId

Returns the perPlayerData structure for a player in a given match. Missing game slots are filled with default 0/false values.

Fully populated games 1–3

Password reset flow

Email-based reset with token expiry

Password reset uses a UUID token stored on the user object with a 1-hour expiry. The flow is:

  1. User submits email to POST /requestPassword.
  2. Backend generates resetToken, stores it on the user, and sends an email with a link to GET /resetPassword/:token.
  3. The HTML form POSTs back to /resetPassword/:token.

Request password reset

POST
/requestPassword

Idempotent: always returns success to avoid leaking whether an email exists.

email No account enumeration

Show reset form

GET
/resetPassword/:token

Renders a modern HTML page with a password reset form. If token is invalid or expired, shows an error page.

HTML response Token expiry: 1 hour

Handle reset submission

POST
/resetPassword/:token

Validates token + expiry, hashes the new password, and clears the token.

password (≥ 8) resetToken cleared
